## このプロジェクト特有の重要ルール
- Codex は進行役であっても、自身の担当成果物以外を確定成果物として作成・更新してはならない。
- `state_compact.json` の `現在ステップ` と `現在の担当エージェント` が示す担当範囲を優先し、担当外フェーズの成果物編集を禁止する。
- 次工程へ進む前に、前工程の成果物作成者と承認状態を確認し、条件未達なら停止して Human に報告する。
- 編集前には「現在フェーズ」「担当エージェント」「これから更新するファイルが担当範囲内か」を必ず確認し、1つでも外れたら編集しない。

---

## 禁止事項（このプロジェクト特有）
- Codex が `docs/DESIGN.md` を正式成果物として新規作成・更新すること
- Claude による設計書未作成の状態で、Codex が `state_compact.json` を `design completed` として進めること
- `承認.設計承認 = false` のまま Codex が実装ファイルや `docs/PATCH.yaml` の作成に着手すること
- フェーズ境界を越える編集を、Human への確認なしで行うこと

## マルチAIエージェントの役割分担と指示（このプロジェクト固有）

本プロジェクトでは、Codex を司令塔兼実装責任者とする**動的ルーティング方式**で開発を行う。
各AIエージェントは固定順で必ず登場するのではなく、Codex が仕事の種類、難易度、検証コストに応じて必要時のみ選定する。

### タスク難易度分類ルール

| 判定軸 | 0点 | 1点 | 2点 |
|--------|-----|-----|-----|
| 変更範囲 | 単一ファイルまたは局所 | 複数ファイルだが局所 | 複数領域または構成全体 |
| 不確実性 | 既知手順で実施可能 | 一部調査や確認が必要 | 前提不明または新規性が高い |
| 外部依存 | なし | 1つあり | 複数または不安定 |
| 実行リスク | 低い | 中程度 | 高い |
| 検証コスト | ローカル確認のみ | 環境確認が必要 | 実機や導入先確認が必要 |

- **軽い**: 合計 0〜3点。Codex 単独で処理する。
- **中くらい**: 合計 4〜6点。Codex + Gemini または Claude のどちらか1つを使う。
- **重い**: 合計 7点以上。Codex + Gemini + Claude を使い、明示的な計画・検証を行う。
- Codex は作業開始直後に、まず **分類のみ** を行い、`state_compact.json` に `タスク分類` / `判定理由` / `使用予定AI` を記録する。

### AIごとの役割

* **Codex**
    * **役割:** 一次分類、タスク重量判定、リサーチ要否判定、担当AI選定、結果統合、実装、ローカル実行、環境検証、実機検証主導、再実行判断。
    * **指示:** 作業後は `docs/PATCH.yaml` を作成・更新し、参照した入力、変更内容、検証予定、未対応事項を明記すること。
* **Gemini CLI**
    * **役割:** 最新情報、公式仕様、API、ライブラリ、制約、料金、手順などの Web リサーチ。
    * **指示:** Codex が「リサーチ必要」と判断した場合のみ `docs/RESEARCH.yaml` を作成・更新すること。独自の設計確定や実装は行わない。
    * **⛔ 無断での自律リサーチ代替は禁止:** Codex は Human の明示指示なしに自分でリサーチ結果を確定成果物として作成してはいけない。ただし、Human が明示的に「Codexがリサーチすること」を指示した場合は、Codex が直接 `docs/RESEARCH.yaml` を作成してよい（その場合は `作成者: Codex（Human明示指示による）` と明記すること）。
* **Claude**
    * **役割:** 計画、設計、責務分離、レビュー、静的検証、設計不整合分析。
    * **指示:** 設計が必要な場合は `docs/DESIGN.md` を作成・更新し、静的検証を行う場合は `docs/VERIFY.md` を更新すること。実装コードの直接編集は行わない。
* **Human / としあきさん**
    * **役割:** 承認境界の判断、実機操作の補助または代行、最終承認。

### 推奨実行形態
- 標準運用では、**Codex 本体が司令塔を兼ねる**。
- Human は要件や補足情報を Codex に渡し、Codex が必要な調査・設計・実装・検証の順序を判断する。
- リサーチは常時必須ではなく、Codex が必要と判断した場合のみ実施する。
- 設計やレビューが必要な場合は、Codex が Claude に依頼する。
- `orchestrator.py` は使用しない。状態遷移の管理責任は `state_compact.json` を更新する Codex が持つ。

### Codex が最初に行う判断
- タスク分類（軽い / 中くらい / 重い）
- リサーチ要否
- 実機検証要否
- 使用するAIの候補
- 初回の検証方針（静的検証 / 環境検証 / 実機検証）

### Codex が各フェーズで必ず守るルール

**分類・リサーチ判定:**
- Codex は作業開始時に、まず**分類のみ**を行い、その後で実装や調査へ進むこと。
- リサーチが不要な場合、`docs/RESEARCH.yaml` が未作成でも異常扱いしない。
- リサーチが必要な場合のみ、Codex は最適な担当AIを選定して `docs/RESEARCH.yaml` の整備を依頼する。

**設計フェーズ:**
- 中くらい以上、または設計不確実性が高い案件では、Codex は Claude に設計またはレビューを依頼すること。
- `docs/DESIGN.md` が必要な案件で未整備のまま実装を開始してはいけない。
- **例外:** Human が明示的に「Codexで設計してください」と指示した場合は、Codex が設計書を作成してよい。その場合は `作成者: Codex（Human明示指示による）` と必ず明記すること。

**検証フェーズ:**
- 実装後、Codex は最低でも環境検証を実施すること。
- 静的検証が必要な案件では Claude に `docs/VERIFY.md` 更新を依頼する。
- 実機検証が必要な案件では、Codex 主導で実施し、必要に応じて Claude にログ解析を依頼する。
- **例外:** Human が明示的に「Codexで検証してください」と指示した場合は、Codex が `docs/VERIFY.md` を直接作成してよい。その場合は管理情報の `作成者` に `Codex（Human明示指示による）` と必ず明記すること。

### docs 契約（固定）
- `docs/RESEARCH.yaml` は必要時のみ作成し、作成する場合の必須キーは purpose / facts / unresolved / design_handoff（YAML形式・AI間受け渡し専用）
- `docs/DESIGN.md` の必須章: 管理情報 / 目的 / 要求事項 / 制約 / 実装対象 / 非対象 / 変更対象ファイル / 実装手順 / テスト観点 / リスク / codexへの実装指示
- `docs/PATCH.yaml` の必須キー: design_id / result / reason / summary / changed_files / verify_request（YAML形式・AI間受け渡し専用）
- `docs/VERIFY.md` の必須章: 管理情報 / 検証対象 / 検証結果 / PASS / FAIL 判定 / 発見した問題 / 原因推定 / 戻し先 / 修正提案 / 再検証条件
- `docs/VERIFY.md` の管理情報には、`検証種別` / `判定コード` / `失敗分類` / `戻し先コード` を必須とする

### 秘密情報と承認境界
- 実パスワード・APIキー・秘密鍵・接続文字列の実値は `SECRETS_LOCAL.md` または `.env` にのみ置く。
- 各AIの参照範囲は `ACCESS_POLICY.md` に従う。
- 本番変更、削除、上書き、認証情報利用、外部公開設定変更は、ユーザー承認がある場合のみ実行する。

## フェーズ運用ルール（AI最適化）

### 基本方針
- **1セッション = 1フェーズ**を原則とする。
- フェーズは `PROJECT.md` の「現在の作業フェーズ」で宣言し、セッション中は変更しない。
- ただし、次工程の選定は Codex がタスク分類と検証状況を見て判断する。
- フェーズを跨ぐ作業が発生した場合は、一旦作業を止めてユーザーに確認する。

### フェーズ定義
| フェーズ | AIが行うこと | AIが行わないこと |
|----------|-------------|-----------------|
| 調査 | 情報収集・分析・報告 | コード変更・設定変更 |
| 実装 | コード生成・ファイル作成 | 本番環境への適用 |
| 修正 | バグ修正・設定修正 | 新機能の追加 |
| 検証 | テスト実行・結果確認 | 本番への反映 |
| 運用 | 監視・定期メンテ | 大規模な変更 |

## AI情報最小化ルール（このプロジェクト固有の補足）

### ログ渡しのルール
- AIには **`artifacts/summary.txt` / `artifacts/errors.txt` のみ**を渡す。生ログ・フルログは渡さない。
- ログ収集には `collectdiag` スキルを使用する（生ログを直接貼り付けない）。
- コンテキスト再送信禁止：一度渡した情報は再送しない。変更差分のみ伝える。

### 状態管理のルール
- `state_compact.json` を状態の主軸とする。
- `state_compact.json` がある場合、`current_status.md` の再読み込みは不要。
- AI切り替え・スレッド切り替え時は `state_compact.json` を最初に渡す。
- `state_compact.json` はメモではなく状態機械として扱い、`現在ステップ` / `次ステップ` / `失敗分類` / `再実行可否` を更新する。
- Codex 主導の動的ルーティング運用時も、`現在の担当エージェント` と `次のアクション` を必ず最新化する。
- あわせて `タスク分類` / `判定理由` / `使用AI` / `research_required` / `検証段階` / `実機検証要否` を更新する。

### FAIL分類と戻し先ルール
| 失敗分類 | 意味 | 戻し先 |
|----------|------|--------|
| `design` | 調査不足・設計矛盾・前提未定義・静的検証失敗 | Gemini または Claude |
| `implementation` | 実装不足・コード不備・PATCH不整合 | Codex |
| `environment` | 環境依存・外部サービス・権限不足・環境検証失敗 | Codex または Human |
| `device` | 実機依存・導入先依存・実機検証失敗 | Codex または Human |
| `approval` | 承認待ち・判断待ち | Human |
| `unknown` | 自動判定不能 | Human |
| `none` | 問題なし | None |

### 再実行ルール
- `失敗分類=implementation` かつ `再実行可否=true` の場合のみ、Codex への再実行候補とする。
- `失敗分類=design` は調査または設計の補強完了まで次フェーズへ進めない。
- `失敗分類=environment` は環境要因と実装要因を切り分けてから再試行する。
- `失敗分類=device` は実機依存条件、導入手順、対象機差分を整理するまで自動再試行しない。
- `失敗分類=approval` は Human 判断が入るまで自動再試行しない。
- `max_step_count` を超える連続試行は禁止し、Human に停止報告する。

### 検証段階ごとの役割
- **静的検証:** Claude が担当する。差分レビュー、設計整合性、責務分離、テスト観点妥当性を確認する。
- **環境検証:** Codex が担当する。ローカル実行、ビルド、単体テスト、疎通確認、ログ確認を行う。
- **実機検証:** Codex 主導で実施する。Windows アプリ、サーバー、ESP32、Excel/VBA、自動化対象など、実際の導入先での確認を含む。
- FAIL 時は、どの検証段階で失敗したかを `docs/VERIFY.md` と `state_compact.json` に明記する。

## ログ・運用ルール

| ファイル | 用途 | 更新タイミング |
|----------|------|----------------|
| `state_compact.json` | AI向け最小状態（主軸） | 作業単位完了ごと |
| `current_status.md` | 人間向け現状サマリー（上書き） | 状態変化時 |
| `work_journal.md` | 人間向け作業ログ（追記専用） | 作業単位完了ごと |
| `CHANGELOG_LOCAL.md` | 影響が残る変更の台帳（追記） | 設定/運用変更時 |
| `<meta>/GLOBAL_CHANGES.md` | 横断影響ログ | 横断影響が生じた時 |

### どこに書くか（判断ルール）
- AIへの状態引き継ぎ：`state_compact.json`
- 人間向けの現状確認・更新：`current_status.md`
- 直近の作業状況・次アクション：`work_journal.md`
- 影響が残る変更（再現性/ロールバック必要）：`CHANGELOG_LOCAL.md`
- 環境全体に影響（FW/DNS/OS/共有資源）：`<meta>/GLOBAL_CHANGES.md`

## 成果物漏れ防止ルール（このプロジェクト固有）
- 設計フェーズを完了する場合、必要なら `docs/RESEARCH.yaml`、必要なら `docs/DESIGN.md` の作成または更新を必須とする。
- 実装フェーズを完了する場合、`docs/PATCH.yaml` の作成または更新を必須とする。
- 検証フェーズを完了する場合、`docs/VERIFY.md` の作成または更新を必須とする。
- `docs/PATCH.yaml` には、対象設計ID、変更ファイル、検証依頼を必ず含める。
- 実装完了報告の前に、`state_compact.json` と `work_journal.md` を必ず更新する。

### Codex オーケストレーション時の承認ルール
- Human から Codex へ要件、調査結果、または「この設計で進めてよい」という承認が渡された場合、Codex はそれを次フェーズ着手の入力として扱ってよい。
- ただし、本番反映、削除、上書き、認証情報利用、外部公開設定変更は従来どおり Human の明示承認が必要とする。
- Claude は設計、静的検証、レビュー、解析の補助に使ってよいが、最終的なコード編集責任は Codex が持つ。
